**上传流程**：
1. 前端 FormData → 后端接收文件
2. 重命名为 `uuid[:12]_title[:40].ext`
3. 存入 `data/materials/<course_code>/`
4. 创建 Material 数据库记录
5. 可选尝试 git commit（`git_storage.py`）

**下载流程**：
1. 根据 material.id 查找文件路径
2. 递增 `download_count`
3. 返回 `FileResponse`（直接读文件流）

### 2.4 数据模型

```
Material 模型:
├── course (FK → Course)
├── title             # 资料标题
├── teacher           # 任课教师（选填）
├── material_type (FK → MaterialType)  # PDF/笔记/试卷等分类
├── description       # 描述
├── file_name         # 原始文件名
├── file_path         # 存储相对路径
├── file_size         # 字节数
├── file_type         # pdf/docx/pptx 等（冗余字段）
├── uploader_name     # 上传者昵称（字符串，非 FK）
├── download_count    # 下载次数
├── is_approved       # 是否审核通过
└── created_at        # 上传时间
```

注意：`uploader_name` 是**字符串字段**而非外键指向 User。这意味着：
- 用户改名后历史上传记录不会更新
- 无法通过 User 实例查询其上传的所有资料
- 个人上传管理需要额外的反向关联

### 2.6 前端文件管理器风格分析

当前 Explorer 视图的交互流程：

```
homeView → 选择"通识课/专业课" (网格卡片)
         → 分类节点 (列表) → ... → 课程节点 (列表)
                                  → 文件表格 + 翻页
```

- 使用**面包屑导航**而非侧栏树形导航
- 一级（课程类型）用方形卡片网格
- 二级以下（分类/学院/课程）用紧凑列表
- 文件层用表格，支持列折叠和水平滚动
- 整体设计与页面设计语言一致（已应用 OKLCH + 圆角 + Noto 字体）

## 三、综合评估与建议

### 3.1 可用性得分

```
用户模块  ★★★☆☆   (3/5)
  - 基础认证流程完整
  - 但缺乏用户中心、密码管理等核心功能

文件管理  ★★★★☆   (4/5)
  - 浏览/上传/下载/搜索/排行 链完整
  - 缺少在线预览和用户管理入口
```

### 3.2 推荐下一步设计方向

按优先级列出：

1. **🔴 用户个人中心页面** — 查看个人信息、修改密码、查看上传记录
   - 影响面：用户模块缺口最大
   - 建议：设计独立的 `/profile` 页面或侧面板

2. **🔴 文件在线预览** — 特别是 PDF 内嵌查看器
   - 影响面：直接影响用户体验闭环（下载后查看 vs 直接查看）
   - 建议：使用 `<iframe>` 或 PDF.js 实现

3. **🔴 用户上传管理面板** — 在自己的 profile 中管理上传的文件（删除/编辑）
   - 依赖：需要先把 `uploader_name` 字符串改为 FK 关联 User，或通过 `Material.objects.filter(course__code__startswith=...)` 变通
   - 建议：设计独立的"我的上传"视图

4. **🟡 密码修改功能** — 前端 API + 页面入口
   - 建议：在个人中心中提供

5. **🟡 文件分类标签增强** — 多标签系统 vs 当前的单 MaterialType
   - 建议：保持简单，现有 MaterialType 够用阶段可暂缓

### 3.3 设计注意事项

- 所有新 UI 必须**先调设计 skill**（`/frontend-design` → `/web-design-engineer`）再写代码
- 保持与现有 OKLCH 色彩体系一致（`--brand-hue` 统一调性）
- 避免"AI 模板风"（浅蓝按钮、圆角卡片、灰色渐变）
- 当前项目审美锚点偏向**学术编辑风**（Noto Serif 标题、暖调藏蓝、奶油底色、宽松间距）
